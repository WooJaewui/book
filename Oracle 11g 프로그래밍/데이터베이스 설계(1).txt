 # 데이터베이스 구축 단계.

요구 사항 분석 -> 논리적 모델링 -> 물리적 모델링 -> 구현( 오라클 )



 # 요구 사항 분석.

설명.
프로그램 방향을 정하고 범위를 한정 짓는 것.

분석 사항.
시나리오 작성.
사용자나 관리자들의 면담, 주요 문서를 통해 시스템 개발 목표 확인.
요청자의 의도를 파악해서 요구되는 정보 확인.



 # 논리적 모델링.

설명.
데이터 모델링을 시작하는 단계이다.
관계형 데이터는 E-R 모델링 등을 통해 작성한다.

용어.
entity : 필요한 정보에 해당되는 주요 대상. ( 속성들을 가지고 있다 )
attribue : 엔티티를 설명하거나 제한하는 특성이나 성질 등의 세부 항목이다.
relation : 엔티티 사이의 연관성을 표현하는 것이다.
Cardinality : 하나의 관계에 실제로 참여할 수 있는 인스턴스의 수. 1:1 , 1:N, N:M
ERD Entity - RelationShip Datagram : 엔티티 관계 모델링 표현하는 방법. ( 도구에 따라 다름 )


 # 엔티티 entity.

엔티티 도출.
요구 사항 정의서에서 명사를 찾아서 표시한다.
명사들 중에 엔티티 이외에 것들을 분리한다. ( 중복된 명사 제거, 명확하지 않은 명사 제거 )

주 식별자.
엔티티에 소속된 인스턴스들을 구분하는 기준이 되는 속성. ( 기본키 )
하나 또는 여러 개의 속성으로 만든다.



 # 관계 relation.

관계 도출.
두 엔티티 타입 사이의 관계를 명확히 파악해야 도출 가능하다.

엔티티 관계 모델링.
ERD를 통해 표현한다. ( IE표기방법 - Crow foot )

외래 식별자.
부모 엔티티의 기본키를 자식 엔티티의 속성으로 가져다가 쓰는 것을 의미한다.
식별 관계 : 부모의 기본키가 자식 테이블에 기본키 또는 후보키인 경우. ( 실선으로 표시 )
비식별 관계 : 부모의 기본키가 자식 테이블에 일반 속성인 경우. ( 점선으로 표시 ) 



 # 용어 사전 data dictionary.

설명.
논리적 / 물리적 데이터베이스 설계시 사용되는 용어들의 의미를 정의해 놓은 문서.
용어는 엔티티(테이블) 이름, 속성(칼럼) 이름을 의미한다.

의미.
설계자가 테이블 / 칼럼을 사용할 때 혼란을 방지하기 위함. ( 설계도 역할 )

규칙.
카멜방식을 사용. 'product_name'
자주 사용하는 너무 긴 용어는 약어를 정의하여 사용.



 # 물리적 모델링.

설명.
논리적 모델링 과정의 산출물을 바탕으로 설계합니다.
엔티티, 관계, 속성, 식별자들을 테이블, 칼럼, 키 등으로 변환하는 과정이다.
